# RFC: Hybrid Workspace & Native Agent CLI Support

## Context & Problem Statement
Currently, NeuroPlasticity deep-copies a user's entire project directory into an ephemeral scratch folder for every epoch. It also relies on hardcoded pre-built container images.

This creates critical friction for testing real-world Agent CLIs:
1. **Auth & Identity:** Tools like `opencode`, `claude`, and `github-copilot-cli` require complex OAuth device flows, keyrings, or environment variables to function. Ephemeral sandboxes break these flows.
2. **Performance:** Deep-copying large repositories (e.g., with `node_modules` or `.git`) on every epoch is prohibitively slow.
3. **Portability:** Users must manually construct complex `base_image` containers to test specific agent versions.

## Proposed Architecture: The Hybrid Workspace

We propose redesigning the sandbox engine to support **Dynamic Dockerfile Builds**, **Read-Only Credential Mounts**, and a **Split Read/Write Workspace**.

### 1. Dynamic Sandbox Building
NeuroPlasticity will natively build containers from a local `Dockerfile` before executing the epoch loop, eliminating the need for pre-published images.

### 2. The Split Workspace Paradigm
Instead of copying files, the engine will leverage Podman/Docker volume mounts:
*   **`/project` (Read-Only):** The user's entire host repository (`$PWD`) is mounted as `:ro`. The agent can analyze the full context instantly with zero copy overhead and zero risk of destructive modifications.
*   **`/workspace` (Read-Write):** A tiny, ephemeral scratch directory is created on the host and mounted as `:rw`. This is where the agent is instructed to write its outputs (e.g., `refactored.py`, `report.json`), and where the bash evaluators will grade the results.

### 3. Native Agent CLI Support (Zero-Config Auth)
To seamlessly support frontier agent CLIs without configuring API keys in tests, NeuroPlasticity will support explicit, Read-Only host mounts. This allows the containerized agent to "wake up" and securely inherit the user's existing host authentication state.

## Schema Updates (`plasticity.json`)

The `sandbox` object will be radically expanded to support these paradigms:

```json
"sandbox": {
  "engine": "podman",
  
  // 1. Dynamic Building (Replaces base_image)
  "build": {
    "dockerfile": "tests/Dockerfile.agent",
    "context": "."
  },
  
  // 2. The Hybrid Workspace
  "workspace": {
    "project_mount": "/project",  // Maps $PWD as Read-Only
    "scratch_mount": "/workspace" // Maps ephemeral dir as Read-Write
  },
  
  // 3. Zero-Config Auth Mounts
  "mounts": [
    {
      "source": "~/.config/gh",
      "target": "/root/.config/gh",
      "readonly": true
    },
    {
      "source": "~/.config/opencode",
      "target": "/root/.config/opencode",
      "readonly": true
    },
    {
      "source": "~/.claude.json",
      "target": "/root/.claude.json",
      "readonly": true
    }
  ]
}
```

## Out-of-the-Box Support (The Gymnasium)

To ensure developers can immediately start optimizing their favorite tools, NeuroPlasticity will ship with a `templates/` directory containing out-of-the-box `plasticity.json` and `Dockerfile` pairs for the big three CLIs.

### Template 1: OpenCode
**`Dockerfile`:** Installs the required rust/node binaries.
**Mounts:** `~/.config/opencode`, `~/.config/gh`

### Template 2: Claude Code
**`Dockerfile`:** `FROM node:20-slim`, `RUN npm install -g @anthropic-ai/claude-code`, sets `CLAUDE_NON_INTERACTIVE=1`.
**Mounts:** `~/.claude.json`

### Template 3: GitHub Copilot CLI
**`Dockerfile`:** `RUN npm install -g @githubnext/github-copilot-cli`
**Mounts:** `~/.config/github-copilot-cli`

## Success Criteria
1.  **Execution Speed:** Epoch turnaround time drops to near-zero as `copy_dir_filtered` is deleted from `src/runner.rs`.
2.  **Safety:** A malicious or hallucinating prompt generated by the Meta-Optimizer cannot physically delete files on the host repository due to the strict `:ro` mount on `/project`.
3.  **Auth:** A user can run `neuroplasticity` against a Claude Code template and Claude will successfully authenticate and execute using the host's existing session token, without the user ever touching an API key.
